Autenticação

Autenticação é o processo que verifica se o indivíduo, entidade ou aplicação informa quem é quem pretende ser.

Boas Práticas de Autenticação

  • ASVS (Application Security Verification Standard)
  • OWASP Cheat Sheet
  • Github - OWASP/ASVS

Ao fazermos a modelagem de ameaças, precisamos analisar todos os pontos de entrada. Isso independente de ser uma aplicação para um usuário final, membro do time, administrativo ou qualquer outro. Nesse caso, estamos com um problema clássico: as credenciais compartilhadas são um problema.

A pergunta fundamental é: "Como sei se a pessoa que está acessando um recurso é ela mesma, de forma verídica e identificável?". Esta é a pergunta que tem que ser respondida ao validar a autenticação nas funcionalidades.

Autorização

Boas práticas para lidar com autorizações

  • Authorization Cheat Sheet
  • Github - OWASP/ASVS

Com o usuário logado e com o token de sessão válido, nos perguntaremos:

  • Quais as funcionalidades que o usuário possui acesso?
  • Quais dados eles ou elas podem acessar?

Toda requisição feita para o servidor, mapeie o que podem ser identificadores. Verifique-os em todo o conteúdo da requisição - na URL, nos Headers, no Body. Ao encontrar algum desses identificadores, questione ou teste se outros que você não deveria ter acesso, funcionam.

Por exemplo: se quiser alterar o seu cadastro e visualizar que no corpo da requisição vai o ID de um endereço que é seu, teste o ID de outro endereço que exista, mas que não é do seu usuário.

Lembre-se: testes de autorização visa verificar se temos acesso de leitura e/ou escrita em informações que não deveríamos ter.

Proteção de Dados

  • C8: Protect Data Everywhere
  • User Privacy Protection Cheat Sheet
  • DATA PROTECTION

Ao nos referirmos à proteção de dados, estamos lidando com mais do que apenas o banco de dados da aplicação. Referimo-nos a todas as informações pessoais, dados sensíveis e/ou sigilosos que a organização armazena. Analisaremos nossa funcionalidade de avaliação de produtos e entenderemos onde esses problemas estão ocorrendo.

Primeiras questões a serem levantadas:

  • O time de desenvolvimento tem acesso aos dados dos clientes?
  • O que aconteceria se um funcionário alterasse o preço de um produto em benefício próprio?
  • Se alguém roubar o backup dos dados, quais informações teria acesso?

Analisando Diretamente o banco de dados

  • Quem possui acesso ao banco de dados?
  • A quais dados essa pessoa possui acesso?

Boas Práticas

1. Separar e restringir os tipos de acesso.

2. Cada pessoa que se conectar no banco de dados, deve ter sua própria credencial, com as permissões especificadas.

3. Usar dados falsos em outros ambientes.

4. Criptografar o armazenamento dos dados.

Validação de dados e Paramêtros

  • Input Validation Cheat Sheet
  • C5: Validate All Inputs
  • Improper Data Validation
  • CWE-20: Improper Input Validation

  1. Validar o tipo de arquivo recebido
  1. Sanitizar ou alterar o nome do arquivo recebido.
  1. Definir e validar o tamanho máximo do arquivo.

Lembrando: Todos os campos devem ter validações que buscam restringir o tipo de informação. Por exemplo: se existe um campo de nome, aceite somente letras e alguns poucos caracteres especiais.

Gerenciamento de Erros

  • Improper Error Handling
  • Error Handling Cheat Sheet
  • C10: Handle all Errors and Exceptions
  • 404 - Not Found

Iniciamos com mais algumas questões

  • O que acontece se alguém tentar de forma recorrente enviar dados inválidos?
  • A aplicação vai ser derrubada se eu enviar uma requisição com dados totalmente errados?
  • Se der erro, o que a aplicação mostra para mim?

Nenhum erro deve ser exposto para fora do servidor!

As respostas da API devem conter apenas uma mensagem comunicando sobre o erro, no máximo junto com um código para que a equipe de desenvolvimento saiba a origem desse erro.

Informação importante: Segurança é uma luta por tempo. Nosso objetivo é fazermos de tudo para que os atacantes percam o maior tempo possível e desistam. Não há sistema 100% seguro, e sim sistemas que os atacantes desistem de atacar.

Auditoria e Logs

  • A09:2021 – Security Logging and Monitoring Failures
  • Logging Cheat Sheet
  • A10:2017-Insufficient Logging & Monitoring
  • C9: Implement Security Logging and Monitoring

Primeiramente é importante definirmos a política de auditoria:

  • O que é necessário armazenar?
  • Por qual motivo esse dado é armazenado?
  • Por quanto tempo esse dado ficará guardado?

Não devemos guardar nos logs informações pessoais e/ou sensíveis. Isso inclui que não devemos logar:

  • Dados de contato, como e-mail e telefone.
  • Dados sigilosos, como senhas e tokens de sessão.
  • Informações pessoais e sensíveis, como dados cadastrais.
